# Tradeoffs — 優先順序與刻意不做

> 時間盒：First commit 起算約 120 分鐘。以下是有意識的取捨，而非疏漏。

## 優先做（依序）

1. **Tailwind + 設計 token** — 先把品牌色與密度定下來，後面所有 UI 一致。
2. **資料層（型別 + mock + async api）** — 契約先行，UI 才有依據。
3. **卡片系統（schema/variant）** — 最高權重，是 momo 神韻與架構亮點所在。
4. **首頁組裝** — 用 sections 把卡片牆、下殺區、排行榜拼起來，視覺衝擊最大。
5. **`/search`、`/goods/:id`** — 走通「列表 → 詳情」資料流，證明分層有效。
6. **`/discover`、`/live`** — 以既有 sections / 卡片變體快速組成，展示可重用性。
7. **README + 文件 + 驗證** — 交付品質。

## 刻意不做（與理由）

| 不做 | 理由 |
|---|---|
| 購物車結帳 / 金流 | 與「架構與卡片可維護性」的考核重點無關，CP 值低 |
| 會員登入 / 帳號 | 純前端無後端，做了也是假流程，不加分 |
| 真實搜尋演算法 | 用簡單字串/分類過濾示意即可，重點是資料流不是演算法 |
| SSR / SEO | 評估重點是前端架構，SSR 會吃掉大量時間 |
| 完整 RWD | 只保證桌機 + 一個主要手機斷點，不逐元件雕 RWD |
| 像素級還原 momo | 抓神韻（密度/促銷/色系）即可，逐像素還原不符投報率 |
| 測試覆蓋率 | 不追求覆蓋率，但留 1 個 schema 驗證 + smoke 示範方向（見 doc 03） |
| 真實圖片資產 | 用 placeholder / 漸層佔位，避免版權與抓圖時間 |

## 若時間不足的退守順序

砍的順序由後往前：先砍 `/live` 細節 → `/discover` 細節 → i18n 第二語系內容（保留機制）。
**絕不砍**：卡片系統、資料層分層、README/tradeoff 文件 —— 這些是評分核心。

## 後續演進方向（給長期維護者）

- **接真 API**：只需替換 `mock/api.ts`，UI/store 不動；可加 React Query 風格的快取層。
- **卡片擴充**：variant + schema 模式可演進成「後端下發版型設定」的 server-driven UI。
- **設計系統化**：token 已集中，可抽成獨立 design-tokens 套件供多專案共用。
- **效能**：商品牆導入虛擬捲動（vue-virtual-scroller）、圖片 lazy load 與 CDN。
- **可觀測性**：把 doc 03 的 dev 驗證升級為正式的 error boundary + 前端監控（Sentry）。
